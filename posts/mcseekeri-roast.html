二、"自研系统" Naomi：史上最大胆的 README

它到底是什么

Naomi 的全称是"NAOMI: Aggregate Online Modular Infrastructure"——光看名字你以为是什么分布式操作系统内核项目。

点进去一看：

  • 语言：Nix
  • Topics 标签：dotfiles, nixos-configuration, home-manager
  • 内容：一堆 .nix 配置文件

翻译成人话：这是一套 NixOS 的 dotfiles。

就像你把 .bashrc 和 .vimrc 传到 GitHub 上，然后宣布"我自研了一个操作系统"。

README 的艺术

最精彩的是 README 里的这句话：

NAOMI: Aggregate Online Modular Infrastructure，简称 Naomi，是基于 NixOS 的"定制系统"。

注意那个引号——"定制系统"。连作者自己都心虚到要加引号了。但到了 B 站宣传的时候，引号就神奇地消失了，变成了"自研系统"。

这就好比你在简历上写"曾在世界 500 强企业'工作'"，面试的时候说"我在世界 500 强工作过"——引号是你最后的良心，摘掉引号就是赤裸裸的欺骗。

技术含量分析

让我们看看这个"自研系统"的核心技术栈：

  • flake.nix —— NixOS 社区的标准模板
  • home-manager —— 社区工具，不是他写的
  • disko —— 社区工具，不是他写的
  • stylix —— 社区工具，不是他写的
  • sops —— 社区工具，不是他写的

他做的事情是：把别人写的工具用 Nix 语言组装在一起，然后声称这是"自研系统"。

这就像你用 WordPress 搭了个博客，然后跟别人说"我自研了一个内容管理系统"。或者你用预制菜热了一桌子饭，然后发朋友圈说"今天亲手做了一桌大餐"。

严格来说，他"自研"的部分是：选择了哪些 .nix 文件放在哪个文件夹里。这确实是一种创造性劳动——就像整理书架也是一种劳动一样。但你不会因为整理了书架就说自己"写了一本书"。

三、关闭 Issue：我听不见我听不见

MCSeekeri 在 GitHub 上有 20 个自有仓库。其中只有 5 个开启了 Issue 功能。也就是说，75% 的仓库直接关闭了社区反馈渠道。

这是什么概念？这就像开了一家餐厅，然后：

但讽刺的是，他的代码真的完美吗？一个 4 星的 dotfiles 仓库，连接受建议的勇气都没有，完美在哪？

更讽刺的是，他自己 fork 了别人的项目（RSSHub、lobehub、nixpkgs 等），享受着开源社区的成果，却不允许别人对他的项目提出任何意见。

只享受开源的权利，不承担开源的义务。这大概就是 MCSeekeri 版本的"开源精神"。

四、CE-RAMOS 造谣事件：一篇文章毁掉一个项目

这是 MCSeekeri 最"辉煌"的战绩。

事件经过

CE-RAMOS 是 PE 圈的一个项目，致力于提供"近乎 WTG 般的 PE 体验"。MCSeekeri 发布了一篇文章，声称 CE-RAMOS 携带恶意程序，列举了 Ventoy.exe、Everything.exe、cpuz.exe 等文件"带毒"。

文章的措辞极具煽动性：

"如果是有意为之……那最好仔细检查你的设备是否安全。"

这句话的潜台词是：CE-RAMOS 的开发者故意在系统里植入病毒。

问题在哪

PE 系统打包第三方工具时，由于重新封装和压缩，杀毒软件误报是极其常见的现象。任何做过 PE 的人都知道这一点。Ventoy 本身就经常被杀软误报，这在 Ventoy 的官方 FAQ 里都有说明。

但 MCSeekeri 的文章完全没有提到"误报"的可能性，而是直接定性为"恶意程序"，并暗示开发者"有意为之"。

这不是安全研究，这是造谣。

后果

文章发布后，CE-RAMOS 的作者遭到大规模网暴。最终，作者不堪压力，放弃了项目。

现在访问 ce-ramos.cn，你只会看到六个字：

该项目已停止服务

一个人花了一年多时间打磨到 V2.3 PRO 版本的项目，就这样被一篇造谣文章毁了。

而 MCSeekeri 呢？他还在 GitHub 上安安静静地维护他的 .nix 配置文件，关着 Issue，关着 PR，仿佛什么都没发生过。

最讽刺的对比

让我们做一个简单的对比：

  • CE-RAMOS：一个真正的 PE 系统，从零开始构建，迭代了一年多，有实际用户
  • Naomi：一堆 NixOS 配置文件，4 颗星，自称"自研系统"

做了真东西的人被逼停服了，写配置文件的人还在吹"自研系统"。

这就是 PE 圈的现状：造谣者安然无恙，创造者被迫退场。
